# Contributing

This repository is a specification and planning store. Contributions should improve source accuracy, implementation readiness, legal scope, safety controls, task clarity, or downstream repository linkage without adding proprietary or private material.

## Acceptable Contributions

- Correct stale or incorrect first-party source URLs.
- Add missing public-source citations when a spec already depends on a behavior.
- Clarify manual verification blockers for paid plans, account-only flows, device permissions, hardware, regional availability, regulated sandboxes, or partner integrations.
- Improve implementation planning, data model, API, safety, privacy, test, or acceptance-criteria detail.
- Record downstream implementation links and parity findings.
- Fix ambiguous affiliation language, overbroad clone claims, or legal-scope gaps.

## Prohibited Contributions

Do not submit:

- Proprietary source code, private APIs, reverse-engineered internals, production credentials, private tokens, private account details, or real user data.
- Third-party logos, screenshots, app-store media, copied marketing copy, copyrighted media, paywalled content, or unlicensed datasets.
- Claims that this project or a downstream implementation is official, endorsed, sponsored, certified, or connected to a source app owner.
- Requirements that depend on a real account, paid subscription, native permission, physical device, regional availability, regulated sandbox, or partner approval unless the blocker and evidence status are explicit.

## Source Corrections

When correcting a source:

1. Prefer exact first-party URLs over search results, blog summaries, screenshots, or third-party aggregators.
2. Preserve the difference between public-source evidence, inferred behavior, and hands-on verified behavior.
3. Keep the affected spec's canonical headings intact.
4. Add or update blockers instead of inventing unsupported parity requirements.
5. Update task history when the correction materially changes scope.

## Manual Verification Evidence

Manual verification notes must be privacy-preserving. Record what was verified, when it was verified, what account/device/region constraints applied, and which blocker changed. Do not include private account identifiers, sensitive personal data, screenshots with personal content, payment details, health data, messages, precise locations, or credentials.

If evidence cannot be shared safely, keep the blocker open and describe the access constraint.

## Downstream Implementation Links

Downstream links should point to repositories or docs that preserve:

- Source spec backlink to this repository.
- Non-affiliation language.
- Original-assets requirement.
- Manual verification blockers.
- License and legal-scope notes.

Do not link public downstream repos until their own legal/name/license/content review has passed.

## Review Checklist

Before a contribution is accepted:

- The changed spec or task doc still has one H1 and stable Markdown headings.
- No proprietary assets, private data, credentials, or copied third-party content were added.
- Non-affiliation language remains clear.
- Manual blockers remain explicit where evidence is missing.
- Downstream public visibility is not implied without approval.
